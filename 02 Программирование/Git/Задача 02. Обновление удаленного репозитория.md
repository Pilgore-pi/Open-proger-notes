## Предисловие

Перед тем, как отправлять изменения на удаленный репозиторий, всегда нужно выполнять `git pull`, для получения изменений, которые могли произойти между моментом коммита и локальными изменениями. В противном случае Git не сможет понять, какая версия репозитория более приоритетна, та что находиться на удаленной версии репозитория или локальная версия, поэтому для введения однозначности в процесс синхронизации, нужно всегда делать получение изменений

## Задача

***Сложность**: очень легко*

Выгрузить изменения локального репозитория на репозиторий GitHub. Изменения были внесены в нескольких файлах каталога "02 Программирование/Git/"

## Алгоритм действий

1. Получение изменений удаленного репозитория GitHub
2. Добавление файлов в индекс
3. Коммит
4. Отправка изменений

## Пошаговое руководство

### 1 Получаем изменения с удаленного репозитория

```git
git pull ProgerNotes master
```

При выполнении данной команды, происходит автоматическое слияние версий локального и облачного репозиториев. Объединенная версия репозитория на данном этапе находится **только на локальном диске**.

Фактически, `git pull = git fetch + git merge`

Если никаких различий между репозиториями нет, то будет выведено сообщение подобного характера:

```git
From https://github.com/Hero/ProgerNotes
 * branch            master     -> FETCH_HEAD
Already up to date.
```

#### Потенциальные ошибки

На практике, при попытке получения изменений со слиянием, может произойти ошибка: `fatal: refusing to merge unrelated histories` (отказано в слиянии несвязанных историй). Такое обычно происходит, если осуществляется попытка выполнить слияние веток, у которых нет общего родителя (грубо говоря, слияние двух веток с разных деревьев). По умолчанию такое действие запрещено, чтобы избежать фатальных ошибок невнимательного программиста. Так как у нас учебный пример, то можно отключить этот запрет:

### 2 Добавляем каталог "02 Программирование/Git/" вместе со всем содержимым в индекс:

```git
git add "02 Программирование/Git/"
```

Здесь кавычки необходимы, так как без кавычек первые 2 цифры воспринимаются интерпретатором Git как отдельный аргумент команды

### 3 Делаем коммит

```git
git commit -m "UPDATE git notes"
```

### 4 Отправляем изменения

```git
git push -u ProgerNotes master
```
