
## Задача

**Сложность: _очень легкая_**

Представим ситуацию, вы создали репозиторий на GitHub с единственным файлом `README.MD`. Вы долго карпели над этим файлом, соблюдали форматирование и пытались сформировать лучший текст для понимания большинством людей. Но вот незадача! Вы запушили свой локальный репозиторий на GitHub в ветку `master`, а на GitHub, по умолчанию, используется главная ветка `main`. Теперь у вас 2 ветки: `main` с файлом `README.MD` и `master` с кучей файлов. Можно, конечно, запуллить (`git pull`) измененя GitHub на локальный репозиторий, изменить его в проводнике и запушить (`git push`) обратно, но удобнее и правильнее совершить слияние веток и удалить `main`. Этим мы и займемся.

## Алгоритм действий

1. Получить изменения с GitHub
2. Совершить слияние веток локально
3. Добавить изменения в индекс
4. Закоммитить
5. Отправить изменения
6. Удаленно удалить ветку `main`

## Пошаговое руководство

Получение изменений с GitHub

```git
git pull origin
```

Слияние

```git
git checkout origin/master
git merge origin/main
```

При выполнении мы получим фатальную ошибку `fatal: refusing to merge unrelated histories` (но это не так страшно как кажется). Это все из-за нашей оплошности, когда мы перепутали названия веток. Ветки не имеют общей истории. Git, по умолчанию, блокирует слияние разнородных веток, так как **эта операция может привести к потере данных**. Будем считать, что мы знаем что делаем (всего-то один файл нужно слить).

Выполняем слияние:

```git
git merge origin/main --allow-unrelated-histories
```

Отправляем изменения:

```git
git push origin
```

Удаляем ветку **на удаленном репозитории**:

```git
git push origin --delete main
```

Обратите внимание, локально ветка удаляется по-другому (ссылка на команду)
