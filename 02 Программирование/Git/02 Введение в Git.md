> Git — это распределенная система контроля версий. Git хранит информацию об изменениях между версиями файлов, а не содержит полноценные копии. 

Система была разработана Линусом Торвальдсом в 2005 году для управления разработкой ядра Linux.

Преимущество Git в том, что, если где-то случился сбой, то проект можно восстановить из любой копии, хранящейся на одном из компьютеров.

- Git в настоящее время — это стандарт в системах контроля версий.

- Можно создавать коммиты — "контрольные точки". В один коммит принято добавлять не более 10 изменений.

- Git позволяет создавать ветки, которые являются альтернативными версиями проекта. Всегда существует главная мастер-ветка, которая предствляет основную версию.

- Git поддерживает многопользовательское взаимодействие

>Данные в системе Git хранятся в скрытой базе данных

## Настройка

После скачивания Git с официального сайта и его установки, появиться возможность открыть терминал Git. В нем пишуться управляющие команды.

Чтобы начать использовать Git, нужно зарегестрировать пользователя. Так как коммиты всегда совершаются от имени какого-то пользователя. Настройка имени пользователя и email:

```git
git config --global user.name "my_name"

git config --global user.email "my_email_address"
```

## Создание репозитория

```git
# Переход в каталог, где должен находиться репозиторий
cd <путь_к_проекту>

# Инициализация (создание) репозитория
git init
```

Теперь Git отслеживает изменения файлов проекта. Но, так как репозиторий только что был создан, в нем еще нет кода, то есть репозиторий пустой.

Чтобы добавить файлы проекта в Git, необходимо сделать **`commit`**.

```git
git add .            # Добавление всех файлов каталога в commit
git add -A           # вариант синтаксиса 2
git add --all        # вариант синтаксиса 3

git add *.cs         # Добавление всех файлов с расширением .cs
git add LICENSE.md   # Добавление конкретного файла

# Создание коммита (сообщение обязательно)
git commit -m "<комментарий>"
```

сообщение должно кратко описывать новые изменения в репозитории

**Как происходит добавление в индекс**

> При добавлении в индекс состояние всех файлов фиксируется, а любые изменения в этих файлах игнорируются, до тех пор, пока еще раз не будет выполнена команда `git add`

Git отслеживает состояние файла в трёх местах:

- Рабочий каталог — текущие изменения.
- Индекс (staging area) — подготовленные к коммиту изменения.
- История коммитов.

### Удаление из индекса

Чтобы удалить все файлы из индекса Git (то есть снять их с отслеживания, но оставить в рабочей директории), используй команду:

```git
git rm -r --cached .
```

Здесь:

- `-r` - рекурсивно для всех файлов и папок,
- `--cached` - удалить из индекса, но не удалять физически из папок,
- `.` - текущая директория (весь проект).

https://git-scm.com/book/ru/v2/%D0%9F%D1%80%D0%B8%D0%BB%D0%BE%D0%B6%D0%B5%D0%BD%D0%B8%D0%B5-C%3A-%D0%9A%D0%BE%D0%BC%D0%B0%D0%BD%D0%B4%D1%8B-Git-%D0%9E%D1%81%D0%BD%D0%BE%D0%B2%D0%BD%D1%8B%D0%B5-%D0%BA%D0%BE%D0%BC%D0%B0%D0%BD%D0%B4%D1%8B

* [`git add`](https://git-scm.com/docs/git-add) добавляет файлы в индекс (staging area) для последующего коммита
* `git status` показывает состояния файлов в рабочем каталоге и индексе: какие файлы изменены, но не добавлены в индекс; какие ожидают коммита в индексе
* [`git commit`](https://git-scm.com/docs/git-commit) берёт все данные, добавленные в индекс с помощью `git add`, и сохраняет их слепок во внутренней базе данных, а затем сдвигает указатель текущей ветки на этот слепок. Атрибуты:
	* `-m "<msg>" | --message "<msg>"` — устанавливает сообщение для коммита, при указании нескольких атрибутов `--m`, их сообщения будут совмещены как отдельные параграфы.

Когда следует делать коммиты?

* Когда создан новый функционал
* Добавлен новый блок на верстке
* Исправлены ошибки по коду
* Сохранение кода после рабочей сессии (или дня)
