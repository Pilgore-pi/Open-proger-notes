
> **Ветка в Git** — это указатель на последний коммит в последовательности коммитов. Это позволяет работать над разными частями проекта независимо друг от друга

## Зачем нужны ветки?

- **Независимая разработка**: Разработчики могут работать над разными задачами параллельно без конфликтов.
- **Изоляция изменений**: Изменения в одной ветке не влияют на другие ветки до слияния.
- **Легкость отмены изменений**: Если что-то пошло не так, можно легко откатиться к предыдущей версии или удалить ветку.

## Создание веток

По умолчанию, при создании репозитория, существует единственная ветка `master`, которая является главной

```git
# создание новой ветки
git branch <имя_ветки>

# переключение на указанную ветку
git checkout <имя_ветки>

# Сокращенный вариант выполнения первых двух команд
git checkout -b branch_name
```

Получение списка веток

```git
git branch
```

Удаление ветки

```git
git branch -d <имя_ветки>
```

Слияние веток:

```git
git checkout master
git merge <имя_ветки>
```

Данная команда выполняет слияние текущей ветки (`master`) с созданной нами веткой

## Стратегии ветвления

Существуют различные стратегии ветвления, такие как **Git Flow**, которая включает в себя ветки:

- `main`
- `develop`
- `feature`
- `release`
- `hotfix`
 
Каждая из этих веток имеет свое назначение и позволяет упорядочить процесс разработки

Стратегии ветвления в Git — это набор правил и практик, которые помогают командам разработчиков эффективно управлять изменениями в коде. Вот некоторые из наиболее популярных стратегий ветвления:

### 1. **Git Flow**

**Описание**: Эта стратегия была предложена Винсентом Дрийсеном в 2010 году. Она включает в себя две постоянные ветки: `master` и `develop`. От `develop` создаются ветки для новых функций (`feature/*`), релизов (`release/*`) и исправлений (`hotfix/*`).

**Преимущества**:

- Подходит для проектов с жестким версионированием.
- Позволяет поддерживать несколько версий одновременно.

**Недостатки**:

- Может быть сложной для небольших проектов.

### 2. **GitHub Flow**

**Описание**: Эта стратегия использует только одну основную ветку (`master`), которая всегда готова к развертыванию. Все остальные ветки создаются от `master` и вливаются обратно в нее.

**Преимущества**:

- Простая и гибкая.
- Подходит для частых релизов.

**Недостатки**:

- Может быть нестабильной для крупных проектов.

### 3. **Trunk Based Development (TBD)**

**Описание**: Основная ветка (`trunk` или `master`) всегда стабильна и готова к деплою. Ветки создаются для небольших изменений и быстро вливаются обратно в основную ветку.

**Преимущества**:

- Упрощает непрерывную интеграцию.
- Сокращает время жизни веток.

**Недостатки**:

- Может быть сложным для крупных изменений.

### 4. **Упрощенная стратегия ветвления (Microsoft)**

**Описание**: Использует ветки возможностей для всех изменений. Основная ветка поддерживается в актуальном состоянии через запросы на вытягивание.

**Преимущества**:

- Простая и гибкая.
- Подходит для команд с частыми обновлениями.

**Недостатки**:

- Может не подходить для сложных проектов с несколькими версиями.


Каждая стратегия имеет свои плюсы и минусы, и выбор зависит от конкретных потребностей проекта и команды разработчиков.
